## NAV line-item amounts live nested under `lineAmountsNormal` — verify decoded structure, don't trust path assumptions (2026-06-16)
The connector's `mapLine` picked `lineNetAmountData`/`lineVatRate` directly off the `line` element (matching the original author's doc comment), but NAV Online Számla 3.0 nests them under `lineAmountsNormal` (VAT) / `lineAmountsSimplified` (AAM). Our `pick()` is direct-child only, so EVERY stored line had net/VAT = null for weeks — descriptions rendered, amounts didn't, and nobody caught it until a dogfood user said "I can't see what I bought." **Lesson:** for any external decoded payload, verify the ACTUAL structure against live data (dump one real record) rather than trusting a comment or the happy-path field name; null-on-every-row is the tell that a path is wrong, not that the source omits data. Reinforces the prior "live pull is the real schema validator" learning. **Process win:** the persona dogfood (a non-technical user actually doing the task) found a data-fidelity bug that passed every build + the invoice-level totals were correct — usability dogfooding catches what type-checks and aggregates can't.

## Idempotency keys must canonicalise values across the source↔DB round-trip (2026-06-16)
The ingest dedup key concatenated raw `(opType, timestamp, net)`. It worked in unit tests (in-memory: the exact string we stored is the string we compared) but FAILED in production because Postgres round-trips the same instant/amount in a different representation: `2026-05-18T08:21:52Z` → `2026-05-18T08:21:52+00:00`, and `numeric` comes back from PostgREST as a STRING not a number. So every re-fetch looked "new" → redundant operation rows + a meaningless "modified" count. **Lesson:** any value used as an equality/dedup/idempotency key must be canonicalised (timestamps → parsed UTC ISO; decimals → a single numeric-string form) BEFORE comparison, on both the incoming and the persisted side — never compare provider strings to DB strings directly. The in-memory fake hid it precisely because it skipped the DB's normalisation; a fake that's "too faithful to our own writes" can mask a round-trip bug. Pairs with [the NAV line-amount nesting lesson] — both are "trust live data over assumptions" failures the live pull caught, not the test suite.

## Audit heuristic classifiers on REAL data; one over-generic token poisons a whole category (2026-06-16)
The deterministic categoriser passed 40 unit tests but mis-booked overhead vendors on Tomi's real invoices: OMV fuel → "Telek/engedélyek" because the regulated-price tag "*HATÓSÁGI ÁR" matched a `hatosagi` stem meant for "hatósági engedély". A single over-generic token, invisible in synthetic tests, produced CONFIDENT (0.6) wrong output at scale. **Lessons:** (1) run any heuristic/classifier over real production data and read the actual outputs -- synthetic tests encode what you expect, not what the world sends; confident-but-wrong is worse than unknown→inbox. (2) Some entities are best recognised by WHO they are (supplier name = a known overhead brand), not by content (line text) -- a name-tier beats a content-tier for vendors whose content is generic. (3) Tie categorisation quality to strategy: here it is wedge-defence (the AI-categorisation layer is half of what a builder-side incumbent would need to close our loop), so "wrong green badges" is a moat issue, not UX polish. Pairs with [the live-pull schema/idempotency lessons] -- the same "trust real data over assumptions" thread.

## Persona dogfood catches "technically correct, perceptually broken" — and it clusters on the front door (2026-06-17)
A second dogfood pass (clean-browser, non-technical SME persona) on the PUBLIC demo found a class of issue distinct from data bugs: things that pass tsc + every test + render the right numbers, yet read as broken or untrustworthy to a human. Examples this run: a terv-vs-tény bar chart whose planned bar was near-white (#EDEFF2) so it looked like ONE green bar with one label (the "3-second scan" promise silently failed); correct data shown with the raw NAV enum "300 PIECE" instead of "300 db"; an intentional pipeline project rendering a bare "nincs terv" that read as "this app does nothing"; a one-tap bulk-assign that would blind-miscategorise the invoices whose own suggestion differed. **Lessons:** (1) build/tests verify correctness, NOT perception — only a human-persona pass catches unreadable-but-correct charts, wrong-locale formatting, and empty states that look like failures. (2) Run the dogfood as a *clean-browser first-time visitor on the public-demo path*, because the public demo IS the front door and a returning tester's stale logged-in session can mask the real first-impression (here it showed an empty real-org project and looked like a leak/bug when a genuine visitor lands on the rich demo). (3) When a feature makes a promise ("whole supplier in one tap", "at a glance"), dogfood specifically whether the UI keeps it on messy real data, not just whether it functions. Pairs with [the prior dogfood data-fidelity lesson] — same tool, different failure class (perception vs. data). **(4) A third failure class the dogfood catches that NEITHER tests NOR in-pixel rendering verification can: "renders correct, but it's the wrong/dishonest thing to show."** On the crew-runway page, in-pixel verification confirmed the gap-filler card rendered the right project with the right colour — but the dogfood caught that the suggested "what could fill the gap" project was itself permit-blocked until *after* the crew dried out, so presenting it as a fill was a fake solution that confirmed the owner's dread instead of removing it. No assertion on pixels or data would flag that; only a human asking "does this actually HELP me, or just look like it?" does. The fix was product logic + honest copy (split genuinely-startable fills from blocked "what's coming", and add a real action — a tappable link). Lesson: after a build renders correctly, still dogfood for *truthfulness and usefulness of the content*, not just its presence and appearance. A feature can pass tests, render pixel-perfect, AND still be subtly dishonest.

## Moat = automation/integration DEPTH, not module BREADTH — adversarially verify "nobody does the set" (2026-06-17)
I drafted a moat thesis ("the wedge is a SET of localized recurring-admin hooks; foreign tools won't copy the set") into the positioning doc. The market agent stress-tested it within minutes and found it WRONG: the broadest local incumbent (LiLBuild) already bundles a wide MODULE set (invoice records + internal log + contract-gen + safety docs + plan-vs-actual + post-calc). The correction that survived: nobody has BUILT the government-bound AUTOMATIONS — NAV auto-pull (vs manual entry), real-time NAV-fed plan-vs-actual (vs retrospective post-calc), official e-napló integration (structurally closed API), EFO labor-report. **Lessons:** (1) module BREADTH is cheap and a competitor probably already has it — never anchor a moat on "we have more modules / nobody bundles these"; the defensible moat is AUTOMATION + INTEGRATION DEPTH (the data flows by itself, closed-loop, real-time), which is hard to copy because it's wired to closed government rails. (2) The incumbent that bundles the generic modules is usually the MANUAL status quo you DISPLACE (the "before" in your story), not a set-competitor — frame it that way. (3) Before baking any "nobody does X" claim into positioning, have an independent/adversarial pass verify it against the BROADEST local bundler, not the obvious head-to-head rival. A plausible moat framing that's subtly wrong is worse than none — it sends the build/narrative at the wrong differentiator. (4) SCOPE every "we're the only one who does X" claim to the category where it's actually true: a capability can be a hard moat in YOUR vertical yet a commodity in an adjacent one (here: native incoming-NAV-pull is unique among construction-PM tools but commodity in accounting software — Kulcs/Novitax/MiniCRM all do it). The refute-proof claim names the scope: "no construction-PM tool does NAV-pull wired to the project cost-line, real-time," NOT "nobody does NAV-pull." An unscoped superiority claim is refutable by one counterexample from the adjacent category and torches credibility. (5) Match verification rigor to need: public evidence (moderate-high confidence) is enough to HOLD a claim internally; spend a scarce resource (a competitor trial account, via the human) to reach airtight ONLY when you're about to make the claim publicly — don't over-verify ahead of need. **(6) This scope-the-claim reflex has now fired THREE times** (NAV-pull = commodity in accounting but unique in construction-PM; könyvelő-export = no tool auto-allocates cost-centre, so "cross-walk" not "auto-import"; idle/continuity-planning = enterprise workforce-mgmt owns *utilization-optimisation* for big GCs, but the *single-fixed-crew "runway"/survival frame* for the SME is unowned). The pattern: the unowned slice is almost never the whole category — it's a **narrower frame** (different user / emotion / math / vertical). Treat "we're the only one who does X" as a reflex to scope BEFORE it reaches a doc, not a claim to defend after. The scoped form is more defensible AND usually more true. **(7) Corollary — the HYPED capability becomes table-stakes; the moat is the hard-to-copy localized/integration depth.** For a vertical SaaS, "AI-native" is fast becoming table-stakes (well-funded EU entrants Pillar/plancraft/Construction-AI all claim it with capital, 2025-26), so it is NOT a USP — lead positioning with what's genuinely hard to copy (here: gov-rail-bound localization — NAV-direct cost-pull, e-napló, EFO, the local workflow), not with the buzzword everyone else also has. Same shape as the breadth-vs-depth lesson: don't anchor the story on the capability the whole field advertises; anchor it on the integration depth wired to closed local rails. Reassuringly, the funded field validated the depth moat by NOT having it (all pull cost from accounting/bank-feed, none from a direct tax-authority API).

## Applying DB migrations (DDL) from the agent env when psql / the DB CLI is absent (2026-06-17)
The project documents migrations as applied via `supabase db reset` / the SQL editor, and the live data runners use the Supabase JS client (PostgREST) — which CANNOT run DDL (CREATE TABLE / ALTER / policies). Neither psql nor the supabase CLI is installed in the agent container, and the app has no `pg`/`postgres` dependency. **Technique that works:** install the `pg` client into a TEMP dir (NOT the repo — keeps node_modules / package.json clean: `mkdir /tmp/x && cd /tmp/x && npm install pg`), then a tiny node script connects via the secret `DATABASE_URL` with `ssl: { rejectUnauthorized: false }` and runs the migration file + seed as plain multi-statement queries (pg's simple-query protocol runs `do $$ … $$` blocks fine), then verifies with SELECTs. **Guardrails:** only do this for ADDITIVE, low-risk migrations (new isolated tables, nullable columns, RLS on the new tables) — never an autonomous destructive/altering migration on production financial tables without a human. Source `.secrets` for DATABASE_URL, never echo it. Mirror the existing RLS pattern EXACTLY (read the prior RLS migration first — anon-reads-demo / authed-reads-own, force RLS, the auth_org_ids() policy shape) so the new tables inherit the same tenant isolation. New tenant tables must carry BOTH the project's tenancy column (`company_id`) AND the RLS scope column (`org_id`).

## A visual/chart fix isn't done until you've read the RENDERED output, not build-green (2026-06-17)
Twice a plausible "make the chart readable" fix passed tsc + build + unit tests yet rendered WRONG. The first attempt recoloured the legend + set the planned `<Bar fill={slate}>` — but both bars still painted green, so the legend now promised two colours the chart didn't deliver (worse than the original). A 2nd persona dogfood caught it by reading the DOM (`both bars fill:#27AE60`). Root cause: **Recharts auto-reads a datum-level `fill` field and applies it to EVERY `<Bar>` series, overriding each bar's own `fill` prop** — our `PlanVsActualDatum` carried `fill` (the invoiced status colour), so the planned bar ignored its slate fill. Fix: rename the datum field (`fill`→`barFill`) so the library stops auto-applying it. **Lessons:** (1) build/tsc/unit-tests verify the code COMPILES and the data shape, NOT the pixels — a chart, layout, or colour fix is unverified until you read the rendered result (screenshot, or query the live DOM: `document.querySelectorAll('path.recharts-rectangle')` → fills). Cheap to do (agent-browser `eval`), and it's the only thing that actually closes the loop. (2) Charting libs (Recharts especially) treat certain data-field names as presentation overrides — never name a datum field `fill`/`stroke`/`color` unless you intend every series to use it; give per-series colours an explicit, non-magic key. (3) When a fix "should obviously work" but a user reports it doesn't, believe the user and read the artifact — don't re-assert the code looks right. Pairs with [the perceptual-dogfood lesson] — the dogfood is what caught both render misses.

## Idempotency unlocks cheap, robust scheduling; keep the fetch window small via a watermark (2026-06-16)
Once the ingest became idempotent (digest dedup), the daily NAV auto-sync could move from an expensive "wake the agent every day" model to a **script-headless** one: the schedule_task pre-script runs the sync itself and wakes the agent ONLY on error -> quiet days cost zero LLM invocations, yet the inbox still fills. This is only safe because (a) idempotency makes a timeout/retry a no-op (no corruption), and (b) the per-org watermark keeps each run's window tiny (~4s incremental) -- crucial because the NAV connector re-fetches DATA for EVERY invoice in the window at 1 req/s, so a 35-day backfill is ~70s (over the 30s script budget) while a 1-day incremental is seconds. **Lesson:** make a sync idempotent FIRST, then you can schedule it headlessly and gate agent wake-ups on exceptions only; and always bound the fetch window with a watermark when the upstream cost scales with window size. The crown-jewel creds stay off the public server (Vercel) — the sync runs in the trusted worker, which is WHY a container-side scheduled task (not a Vercel cron) is the right home.

## A read-first wedge's edit path = progressive-disclosure islands on the answer surface, not a separate admin screen; and the manage-trigger should BE the state's action (2026-06-17)
The crew-runway shipped as a clean answer-first page (alarm line → cards sorted worst-on-top). Adding the CRUD edit path risked turning that anti-slop surface into an admin grid — the exact thing the wedge avoided. **What worked:** keep the read page as-is and add the write surface as **per-card progressive-disclosure islands** (collapsed by default; the CRUD only appears when the owner taps "manage"), plus a small launcher for the rare add-parent action. Crucially, **the manage-trigger label is derived from the row's state so the fix lives where the diagnosis is**: a drying-up (red) crew's trigger reads "plan the next work" and opens *straight into the add-form* (the one action that clears its alarm — adding an allocation extends `max(end_date)`); a stale row → "update the end date"; the rest → a quiet "manage". The page doesn't just show a problem and make you go find the editor — the same surface diagnoses AND fixes. **Lessons:** (1) don't bolt a generic admin/CRUD screen onto a read-first wedge; disclose the edit affordances in place, contextualized to each row, so the "useful help not admin burden" north-star survives the addition. (2) Make the primary write affordance the *named action that resolves the row's state*, not a generic "edit" — it turns a dashboard into a tool. (3) Reuse the existing write convention verbatim (here: `requireOwnerOrg` → zod mirroring the DB CHECK constraints → secret-client write with explicit org/company stamping → revalidatePath) so the new tables inherit the already-proven auth/RLS seam instead of inventing one; and gate the whole edit surface on `canEdit` so the public demo stays read-only *by construction* (a write resolves to `demo_readonly`, never lands).

## You can't dogfood a write path that needs real-user auth when the demo is read-only and the real org is consent-gated — stand up a seeded sandbox authed org (2026-06-17)
The crew edit path is structurally identical to two production write actions (so tsc/build + the proven sibling RLS gave high confidence), but the **authed end-to-end write was genuinely untestable in this run**: writing to the pilot's real org needs the human's consent (data-sovereignty rule), and the public demo is read-only (no anonymous path exists to exercise the write UI). So a persona dogfood of "log in → add my first crew → plan work → watch the alarm clear" had nowhere to run. **Lesson:** when a feature's write path is gated behind real-user auth, the public/demo surface is deliberately read-only, AND the only real tenant is the consent-protected pilot, there is NO safe path to dogfood the write flow — so provision a **seeded sandbox authenticated org** (fictional data, a throwaway login) as standing infra. It lets every future write-path get a real persona dogfood without ever touching the pilot's data, and closes the one seam that "renders correctly + compiles" can't. Be honest in the log about the untested seam rather than implying the write path was exercised — "structurally identical to a proven path + builds green" is a real confidence basis, but it is NOT the same as having watched it work. **Realized same day (2026-06-17):** stood up the sandbox org (auth.admin.createUser with the secret key + orgs/org_members insert + the existing provision-company RPC + a few seed projects, no crews) and ran a real persona dogfood logged into the deployed app. It paid off immediately — a write path has its OWN failure class the read-path dogfood and the build could never catch: **trust in the save.** The #1 finding was that after clicking save the form gave no positive confirmation, so a builder unsure it saved would re-enter or abandon — invisible to tsc/tests/pixels, fatal to a planning tool. (Plus: native date-input locale chrome, and a summary banner that only lied once a real zero-allocation row existed — which only an actual add-a-crew flow produces.) Reinforced: a write path deserves its own logged-in persona dogfood, and the sandbox org is the standing infra that makes it cheap + safe — fold the friction and re-verify on the live deploy, the same loop as a read-path dogfood.

## For a digitally-immature SMB ICP the adoption barrier is GUIDANCE, not motivation → assisted/done-for-you onramp wins, self-serve-only loses the majority (2026-06-17)

## A Next.js server action implicitly refreshes its OWN route — so a client component holding optimistic state must be ALWAYS mounted, not conditionally rendered by the server (2026-06-17)

## "Guard checked at read-time" ≠ "guard enforced atomically" — the atomic-claim is its own sibling-audit dimension (2026-06-18)
A follow-up to [the missing-guard sibling-audit lesson]. After shipping a NEW mutating action (`supersede`) on the same evidence core, I applied that lesson's rule (audit the siblings) — and it paid off, finding a whole *class* of race that the earlier pass hadn't: `edit`/`delete`/`supersede` each **had** the status guard, but only checked it at READ time, then issued an **unconditional** write. So a concurrent state change landing between the read and the write slips a mutation onto a record that just left the guarded state — edit/delete a now-`sent` record (whose text the counterparty may be viewing, or whose audit/confirmation trail would cascade away), or two concurrent "correct this" submits producing two divergent replacements. The confirm path had already been made an atomic claim (`WHERE confirmed_at IS NULL` + rows-affected) in the prior red-team, but its siblings were left as read-then-write. **Lessons:** (1) a guard being *present* (checked after a read) is a DIFFERENT property from being *enforced atomically* (in the write's own `WHERE` + a rows-affected check) — audit for BOTH; the second is invisible to "is there a guard?" review and to any single-threaded test or dogfood. (2) the **atomic-claim is itself a sibling-audit target**: the moment ONE mutating action becomes a conditional claim (race-safety win), every status-guarded sibling that's still read-then-write is a latent race — make them all `UPDATE/DELETE … WHERE <guard> … .select()` and check rows-affected; snapshot/insert side-effects go only AFTER a won claim (and for a multi-write action like supersede, insert the new row first so an insert failure leaves the old untouched, then claim-flip and roll the new row back on a lost claim, so divergent versions can never both persist). (3) **a write-MECHANICS refactor (read-then-write → conditional `.update/.delete().select("id")`) MUST be dogfooded on the live DB** — a mock proves your control flow but NOT the driver's return shape, and the whole refactor hinges on the real client returning the affected rows for a conditional write (if supabase-js returned null/empty there, every happy path would wrongly self-report `locked`). The live dogfood (all three mutations end-to-end on the deploy) is the only thing that closes that assumption; same "trust live data over the mock/assumption" thread as the schema/idempotency lessons.

## A moat invariant instructed-in-the-prompt is not enforced — put it in the deterministic layer; and provenance is often a SCOPE/layering decision, not a heavier guard (2026-06-20)

The pipeline had meanwhile polished the table-stakes guards (malformed-JSON tolerance, NaN-confidence, injected-terminal-state) that any platform also ships, while the ONE moat-differentiating guard was unbuilt. Building the differentiator beats polishing the commodity guards.
**How to apply:** for any AI-extraction/inference product: list the invariants that ARE the moat, and verify each is enforced in pure/validation code with a test, not asserted in the prompt. Before reaching for a heavier guard, ask whether scoping or normalizing the input makes the simple guard correct. On the trust path, prefer drop-with-telemetry (log the dropped items) over a looser match. Cross-project: extraction, classification, RAG-with-citations, any "show the source" feature, any moat that depends on the model behaving.

**Generalization (2026-06-21, the reconcile build):** a moat invariant delegated to the CALLER (a contract: "the caller will pass a counterparty-scoped candidate set") is the SAME failure mode as delegating it to the prompt — and a red-team will find it. The pure guard accepted whatever set it was handed. Fix: enforce the identity floor (same-counterparty for any merge; degrade a multi-counterparty set to no-merge) IN the pure guard, independent of how the caller assembles inputs. **The rule extends: the deterministic layer must defend the moat invariant against (a) the untrusted model, (b) a prompt that wasn't followed, AND (c) a caller that assembled inputs wrong — a pure guard that trusts its own caller's contract is not a guard.** Test the guard with a deliberately-mis-assembled input, not just a well-formed one.

## Date-relative UI: compute calendar-day deltas from the user's LOCAL date, not UTC instants — and the bug is INVISIBLE in a UTC dev box (2026-06-22)
**What happened:** the Kept "Commitment Field" SPA labeled urgency ("ma esedékes / holnap / N nap múlva") by rounding the *instant* difference `(dueMs - now)/86.4M`. Dues are stored as UTC-midnight date-markers, so against a real-time `now` the rounded delta drifts by the user's UTC offset at the day boundary: at 23:00 local a "due tomorrow" item rendered "ma esedékes" (due today), and the "MOST" header showed the UTC date, not the user's today. In a product whose entire value is *right-timing*, this is a trust-killer (the ux-critic graded it a ship-blocker; a red-team independently flagged the same math as a preview/commit skew). The fix: treat "a day" as a calendar DATE → a UTC-midnight anchor; **today = the browser's LOCAL `getFullYear/Month/Date`** (the browser knows the user's zone for free), a **due = its stored `getUTCFullYear/Month/Date`** marker; the delta is the exact integer between the two anchors. The reveal: the bug was *invisible* in the dev container because local==UTC there — it only surfaced because that night the container clock happened to sit in CEST, so at 00:30 CEST (UTC still the previous day) the live render proved the fix (MOST = "jún 22" local while `new Date().toISOString()` = "...T22:30Z jún 21").
**Why it matters:** (a) instant-rounding ≠ calendar-day difference — the two diverge by the UTC offset near midnight, which is exactly when "is it due today?" anxiety peaks. (b) A timezone bug in date-relative copy passes every test and every dev session run in a UTC box (the universal CI/container default), then mislabels for every real user not on UTC — a structural blind spot, like the scale-case and the dead-wire blind spots. (c) The browser is the cheapest correct TZ oracle you have; reaching for `getUTC*` to render a user-facing "today/tomorrow" is the smell. (d) Beware introducing a *surface-to-surface* divergence: fixing the web to local-calendar while the server (bot) stays rounded-UTC means the two can disagree at a boundary — the real fix unifies both (give the server a configured TZ), which is its own slice because it changes live behavior.
**How to apply:** (1) For any "today/tomorrow/N days ago/overdue" copy, compute the delta as a calendar-day difference anchored on the *viewer's* local date, never a rounded instant; render the viewer's own "today" in local time, not UTC. (2) Test the boundary explicitly — inject a `now` at 23:30 local in a non-UTC zone and assert the label, OR at minimum run a smoke-check with the container TZ set to the target zone; a UTC-only test will never catch it. (3) When you fix a date computation on one surface, check every *other* surface that shows the same relative date (web vs bot vs email) for consistency, and prefer one shared, configured-TZ source of truth. (4) Keep a date's preview and its commit on the SAME arithmetic — if a drag shows "+2 → jún 23" it must WRITE jún 23, not a separately-computed instant. Cross-project: any reminder/calendar/deadline/"last seen"/relative-time UI, any multi-surface product, any feature whose value is timing.

## A promise on a trust surface must be validated against the WHOLE delivery path, not just the local module's truth (2026-06-22)
**What happened:** Kept's capture confirmation (a pure module, `confirm.ts`) added an honest line stating WHEN a captured item resurfaces -- for a dated item, "Szólok, ahogy közeledik a határidő" (I'll bring it up as the deadline nears). Cross-project: any notification/reminder/SLA/receipt/confirmation copy, any layered system where the component making a guarantee is separated from the component that fulfills it, any "claims = truth" trust product.

## A proactive/recurring-surfacing feature needs a STEADY-STATE governor (re-fire cap + backoff + age-window), not just a cold-start trigger (2026-06-23)
**What happened:** building Kept's `stale-open` trigger (surface an undated open promise that's gone quiet — "you promised X, still open?"), I got the COLD START right (a salience-scaled grace deciding WHEN it first fires) but shipped it to red-team with ZERO discipline on the STEADY STATE. Two defects, both moat-inverting: (1) an undated nudge can never escalate (it never gains a deadline), so on the flat cooldown it would re-fire every 3 days FOREVER until the user acted — the textbook nag that drives uninstall; (2) anchoring "age" on the row's `createdAt` meant a bulk import (every row created at the same instant) would make ALL historical items ripe on the same day → a months-long daily drip about ancient data. The fix was the governor the feature was missing: a per-item re-ask CAP with an ESCALATING gap (3d→6d→12d then quiet), and an age WINDOW `[grace, maxAge]` anchored on the real event time (`evidence.capturedAt`), not the ingest time. The red-team's framing was exact: "the team built precision into the cold-start and zero discipline into the steady-state, which is backwards for a low-noise product."
**Why it matters:** any feature that can REPEAT (a reminder, a digest, a re-engagement ping, a retry, a "you have unread X" badge, an alert rule) has two distinct regimes — the first fire and the Nth fire — and they need separate design. The first-fire logic is the fun part and gets all the attention; the steady-state is where the product either feels respectful or becomes spam, and it's invisible in a unit test (which fires once on a fresh fixture). The two classic steady-state failure modes are (a) flat-cadence repetition with no backoff/cap → "it won't stop," and (b) a clock/age anchored on system time rather than event time → a bulk import or a clock skew makes everything fire at once. A low-noise product (the whole value is "the rare, well-timed ONE") is destroyed by either, and worse, a confirmation that PROMISED the behavior makes the spam read as working-as-designed, not a bug.
**How to apply:** (1) For any feature that can fire more than once, design the re-fire schedule explicitly BEFORE shipping: is there a cap (fires N times then goes quiet)? a backoff (the gap widens)? a terminal silent state the user can re-trigger? "Fires whenever the condition holds" + a flat cooldown = forever-nag. (2) Anchor any age/staleness/recency on the real EVENT time (when the thing was said/happened), never on the row's `createdAt`/ingest time — then test the bulk-import / backfill case explicitly (N items created at one instant with event times spread across history; assert they don't all fire on the same day). (3) Add an upper bound: a thing old enough is no longer "fresh news" — it belongs in a list/archive, not the proactive feed. (4) Unit-test the STEADY STATE, not just the first fire: drive the history forward (item already surfaced k times) and assert backoff + eventual silence. (5) Run an adversarial "what does the Nth fire feel like, and what happens on a backfill?" review on any recurring surface before it ships. (3) The activation metric (acted-on-nudge) records non-saves as saves for seeded items *by construction*, because a seeded item is something the user *just proved they remember*. The cold-start has to be carried by whatever fires AUTOMATICALLY from the minimum seed, even if that's the "boring" mechanism; the magic is a steady-state delight, not an onboarding guarantee. (b) "Manufacture an honest moment fast" is a genuine contradiction (fast+deterministic reads as a scripted replay of the user's own input; slow loses the user). You don't resolve it on the timing axis — you resolve it by finding the real, defensible VALUE the early moment delivers (here: offloading the vigilance, not rescuing the memory), which also tells you what to MEASURE. ## In a deferred-gratification / honest-silence product, deliver the relief payoff as PRESENT-PROOF inside the user's own action — never as a future-promise the system will perform later; and the empty-state BETWEEN actions is the highest-stakes, most-undesigned surface (2026-06-23)
**What happened:** designing Kept's cold-start, my countermeasure for the trust-decay gap (the user offloads their mental list, then waits ~3 days of silence for the first proactive proof) was a future-promise line: "I'll bring the first of these back in a few days." A ux-critic lens killed it cold: a vague future promise ("a few days" reads as *maybe/whenever*) is the weakest possible trust instrument — it's delivered at the moment of LOWEST doubt (day 0, optimism high) and asked to carry trust across 72 hours of RISING doubt, and it proves nothing the system currently holds. The fix was to invert it: deliver the payoff as PRESENT-PROOF inside the user's own action — a single unprompted "I've got it" beat ~1 hour after the offload that reads the captured set back as a *held state* ("mostantól én figyelem helyetted") and grants explicit permission to stop holding it ("elengedheted"). That delivers the relief INSIDE the cold-start window instead of deferring it to a timer that might fire days later (and might fire on a stale item). Two deeper structural finds: (1) for a product whose value is being quiet and proactive, the empty-state BETWEEN actions — the silent first week before any proof — is the single most dangerous surface, and it was the one the design had reasoned about (the trust-decay race) but never actually *designed a screen for* (the whole onboarding was "unwritten prose" — concepts, not surfaces). (2) The fix wasn't new surfaces — the good empty-states already existed (a clean `/lista`); the gap was that NOTHING ROUTED the day-1 user toward them. The first proactive beat is the router.
**Why it matters:** (a) Any product with a gap between user-effort and system-payoff (proactive assistants, scheduled digests, "we'll notify you when…", background processors, anything with deferred gratification) has a trust-decay window in that gap, and the instinct — reassure with a promise about future behavior — is the weakest move, because a promise about the future is unfalsifiable-now and reads as a deferral. The strong move is to manufacture a payoff in the PRESENT out of what the user just did: reflect their own input back as an accomplished state, so the effort feels *already rewarded* rather than *banked against a future the system controls*. (b) Quiet-by-design / low-noise / honest-silence products have a structural blind spot: the team lavishes design on the active moments (the capture, the nudge) and leaves the SILENT stretches — which are most of the lifetime and the highest-doubt moments — as undesigned dead air. "Honest silence" is a principle, not a screen; the silence still needs a designed felt-experience (what the user sees if they look, what told them it's intentional). (c) Empty/between states are usually a *routing* problem before they're a *content* problem — the reassuring surface often already exists; nothing points the user at it at the moment of doubt. (d) Two review lenses catch different failure classes: the red-team (mechanism/correctness) validated the triggers and found a spec hole; the ux-critic (felt experience) found the missing surfaces and the trust-decay copy — neither alone was sufficient, and the felt-experience gaps were INVISIBLE to the mechanism lens because the mechanism was correct.
**How to apply:** (1) Wherever there's a gap between a user's effort and the system's payoff, design a PRESENT-tense proof delivered inside the user's own action — reflect their input back as an accomplished/held state and, where the value is offloading a burden, explicitly grant permission to let go — instead of a future-promise about what the system will later do. A vague time-promise ("soon", "in a few days") is worse than no promise; if you must signal timing, make it conditional and active ("if X happens, I'll Y"), not a countdown. (2) For any quiet-by-design / low-frequency / honest-silence product, explicitly design the BETWEEN-state and the empty-state as first-class surfaces — what the user sees and feels in the silence, and one (exactly one — a second becomes the noise the product rejects) unprompted beat that frames the silence as intentional discipline, not death. (3) Treat empty/doubt states as routing first: check whether a reassuring surface already exists and the real gap is that nothing leads the user to it at the right moment. (4) Run BOTH a mechanism lens (red-team) and a felt-experience lens (ux-critic) on a design before it ships to the owner — they catch disjoint failure classes, and a correct mechanism makes the UX gaps invisible to the correctness lens. (5) When you write "the user will see X" in a design and X is a concept not a concrete screen/message, that's a tell you've left a surface undesigned — name the actual copy/state, especially for the unglamorous between-moments. Cross-project: any proactive/assistant/notification/digest product; any deferred-gratification or background-processing flow; any low-noise/quiet-by-design product; any onboarding with a silent gap before first value.

## Giving an isolated sandbox agent a privileged HOST capability (deploy) safely = shared-mount + a fixed host watcher + a marker-file protocol — no shell, no secret exposure; and red-team infra for the bot-down / secret-leak paths the happy path can't show (2026-06-23)
**What happened:** the #1 dev-friction on Kept was that my committed code wasn't live — the host CLI that deploys is ad-hoc (Tomi opens/closes it), so every staged commit sat "committed but not running" for days AND I was blind to whether it was running. Tomi asked me to solve it. The build context is already a SHARED MOUNT (host repo == my sandbox repo, same disk), so I used that as the channel: I write a `request.json` marker into a gitignored+dockerignored control dir; a fixed host watcher (systemd) polls it, runs a test-gated build → container swap → health-check → **auto-rollback**, and writes `health.json`/`status.json` back for me to read. I get NO host shell — only "write a marker, read health." Secrets are lifted from the *running* container by full var name, never logged, never surfaced to me. The whole thing is files I author + ONE human paste-prompt to install the watcher once.

The agent's blast radius is exactly the marker schema, nothing more. (b) The visibility half is as valuable as the action half: "committed ≠ live, and I can't see which" is a silent productivity tax across many sessions; a continuously-written `health.json` with a `live_matches_head` flag turns "did my code ship?" from a guess into a read. (c) Infra like this earns a dedicated adversarial red-team BEFORE it reaches the human, and the findings cluster on classes the happy path structurally can't surface: **bot-down-with-no-recovery** (rollback image missing/stale, destructive-first swap, a failed deploy re-looping every poll and *re-killing* the service — the flap loop), and **secret leakage through an unexpected channel** (the Telegram bot token lives IN the API URL, so a logged fetch-error leaks it into the very health file the agent reads — "never logged" is an assumption, not an enforcement). The red-team caught three real ones I'd have shipped.
**How to apply:** (1) To give a sandbox agent a host capability: shared-mount marker protocol + fixed host watcher + narrow verb; secrets lifted host-side from the running process, never passed through the agent, never logged. Make the control dir gitignored AND dockerignored so it can't dirty the tree or bloat the image. (2) Make the trigger idempotent and the watcher **claim each request id BEFORE running** (at-most-once), gated on a file the *watcher* owns — never on the deploy script's best-effort status write — so a failed/stuck deploy can never flap the live service on the poll interval. (3) Any deploy/swap MUST: gate on tests, build a *candidate* image (keep the current one tagged for rollback, captured from the RUNNING container so a stale `:latest` can't strand you), health-check on the service's real anchored ready-banner (not a loose substring), and auto-rollback. Refuse to build a dirty tree or an unpinned commit. (4) Before infra reaches the human, red-team it specifically for (i) every path where the swap kills the service and neither candidate nor rollback comes up, and (ii) every channel a secret could escape into that the agent or a log can read — and remember provider tokens-in-URLs (Telegram) defeat "I never log env vars." Redact secret-shaped strings (`bot\d+:[\w-]+`, `sk-ant-[\w-]+`) from any log tail you expose. Same family as [trust live data over assumptions] (here: the red-team trusts the failure modes over my confidence) and the missing-guard / atomic-claim threads (the flap-loop is a missing at-most-once guard on an idempotent trigger).
**Validated in production (2026-06-25):** the human installed the watcher and the FIRST real deploy went end-to-end on the first try — test-gate (603 pass) → candidate build → 6 secrets lifted host-side → swap → anchored health-check → `LIVE ✓`, with auto-rollback armed but unneeded, finishing in ~25s; `live_matches_head` flipped to true and the "committed ≠ live, and I can't see which" tax was gone in one move. The designed pattern is now a proven one — the red-team-found classes (flap-loop, token-in-URL leak) were fixed *before* this first run, so the first contact with production was clean rather than a debugging session. Lesson reinforced: red-teaming infra BEFORE it touches the human is what buys a clean first deploy instead of a live incident.

(c) `\b`, `\w`, `[a-z]`, `.toLowerCase()` (vs locale-aware), naive capitalization, and `\d`-only number parsing are all ASCII-centric primitives that silently misbehave on the exact non-English input the product is FOR.
**How to apply:** (1) Any deterministic parser/normalizer/classifier of human language or human-entered data must be tested in the LANGUAGE(S) AND LOCALE(S) OF ITS ACTUAL USERS, not the author's — make "what does the real user type?" an explicit test-design question, and if the product is non-English-native, the non-English cases are the PRIMARY suite, not an afterthought. Capture the pre-normalization input before any ASCII-centric transform can corrupt it, and prefer Unicode-aware constructs (`\p{L}`, locale-aware case) where the text is human language. (4) The fix's residual imprecision (intra-day phrases collapsed to "today" as a v0 stopgap) is fine to ship deliberately and log — the bar is "never silently wrong/blind," not "perfect." Cross-project: any i18n/L10n product where the dev's language ≠ the user's; any deterministic NLP/date/number/name/address parser; any regex over user-entered or non-English text; any "thorough green suite" on a component whose real inputs the author may not have imagined.

**What happened:** Kept silently surfaced nothing for its dogfooding owner ("it never says anything, not once"). Live instrumentation proved the cause: a day-1 precision gate — a hard-promise must clear 0.85 extraction-confidence to be eligible to surface — was dropping 5 of the owner's 7 deliberately-dictated promises, which scored in [0.5, 0.85). The gate's rationale was sound but built for the WRONG input source: 0.85 was tuned for a future AMBIENT firehose (passively-ingested email, where a false "you promised X" is un-opted-in noise → precision must dominate). But v0 capture is 100% DELIBERATE first-party dictation — the user SPOKE the obligation into the bot and got a "noted, I'll remind you" receipt. For a deliberate capture the "is it real?" question is already answered by the act of recording; the confidence mostly reflects PARSING ambiguity, and the gate's effect was to make the bot CONFIRM a reminder and then never deliver it — a broken promise, which is a far worse trust failure than a marginal nudge the user can dismiss in one tap. The fix moved the precision job off the confidence NUMBER onto the guards that are honest for deliberate capture (the evidence anchor shown, a one-tap dismiss, the ≤1/day budget) and lowered the floor. The second half: I'm a sandboxed operator with NO network/DB access to the live daemon — I'd been GUESSING at the cause across multiple sessions. The unlock was a privacy-safe diagnostic (`src/diag.ts`: counts, buckets, booleans, timestamps ONLY — never any user text) that the deploy runs inside the live container and folds into the status file I can already read. It instantly showed `eligible: 6/11`, the confidence histogram, AND an unexpected second finding (a nudge HAD fired that day, so the ≤1/day budget was the now-binding constraint) — turning a multi-session guess into a one-read diagnosis.
**Why it matters:** (a) A confidence/quality/spam/precision threshold is implicitly calibrated for a population, and the SAME object from a different source belongs to a different population with a different cost-of-error. The decisive variable is often INTENT/opt-in: passively-harvested data (low intent → a false positive is pure noise → gate hard) vs. deliberately-submitted data (high intent → the user vouched for it → recall matters, and the false-positive cost is low because they can correct it). Applying the ambient threshold to deliberate input inverts the precision/recall tradeoff exactly backwards. The right design is provenance-aware (the gate varies by source), not one global constant. (b) In any product that ACKNOWLEDGES an input ("saved", "I'll remind you", "we'll notify you"), a downstream silent filter that later drops that item makes the system a liar — the acknowledgment is a promise, and a precision gate that silences a confirmed item breaks it invisibly. (c) When you operate a system you can't directly observe (sandboxed agent, prod you lack creds for, a customer's box), GUESSING burns sessions and ships wrong fixes. Build observability into the channel you DO control — and make it privacy-safe by construction (counts/buckets/booleans/timestamps, never raw user content) so it's shippable to the most sensitive product. A counts-only snapshot folded into an artifact you can already read (a deploy status file, a health endpoint) converts "I think it's the gate" into "eligible went 6→11, and the budget was already spent" — including findings you weren't looking for.

(2) Audit every place the system ACKNOWLEDGES input against every downstream filter that could later drop it — a confirmation is a promise; a silent gate that falsifies it is a trust bug even when each layer is locally correct. Weight error directions by trust cost: a broken promise (confirmed-then-silent) usually outweighs a dismissable false positive. (3) When you can't observe a system you operate, your FIRST move is to build privacy-safe observability into a channel you control (deploy artifact, health file, log tail) — counts/buckets/booleans/timestamps only, zero user content — before you theorize; it pays for itself in one read and surfaces the findings you didn't think to look for (here, the budget constraint hiding behind the gate). (4) After a fix on a blind system, read the SAME diagnostic to CONFIRM the change took (eligible 6→11) rather than trusting the deploy succeeded. (5) Distinguish the binding constraints: fixing the gate revealed a SECOND limiter (the ≤1/day budget) — a "nothing happens" symptom can have stacked causes, and the diagnostic that proves the first should also expose the next. Cross-project: any confidence/quality/spam/moderation threshold; any product that confirms/acknowledges then asynchronously filters; any reminder/notification/SLA promise; operating any system you can't directly attach to (sandboxed agent, customer-hosted, prod-without-creds); any privacy-existential product needing observability without exposing user data.

## An observability instrument that LIES is worse than none — a generic "did the call succeed (not throw)" wrapper must never stand in for a domain "did the action happen" boolean; and under an alarm, verify the INSTRUMENT before you disclose the incident or ship a fix (2026-06-27)
**What happened:** the day after building Kept's privacy-safe push-heartbeat (a `console.log("nudge PUSHED")` the host watcher tails into `health.json`, so I can see the proactive moat fire without a deploy), my standing health-watch showed a horror: 52 `nudge PUSHED` lines in 4h, one every ~75s, straight through the closed delivery window — i.e. the entire low-noise moat appearing to spam the dogfooding user a message every 75 seconds at midnight, a trust-existential SEV-1. But the code gates (window + budget) both read correct, so instead of reacting I investigated: I built a non-destructive `[DIAG-ONLY]` deploy-note mode that dumps the RUNNING container's timestamped logs + a fresh reasoner snapshot (the sandbox is blind to host logs, and the instrument itself could no longer be trusted). The smoking gun: 52 phantom lines, while a SEPARATE process (`diag.ts`, which reads the real decision path) correctly reported window-closed, budget-exhausted, ZERO real sends. Root cause: the heartbeat read the wrong boolean. The tick wrote `const sent = await best("surfacingTick", () => maybeSurface(...))`, but `best(label, fn)` is a generic transport helper that returns whether `fn` *resolved without throwing* — it ignores `fn`'s return value. `maybeSurface` returns `false` on a silent beat (window closed / all-clear) but does not throw, so `best` returned `true` every beat → the heartbeat logged a phantom push on every silent tick. **No user was ever messaged; the gates were healthy the whole time.** The instrument I built to PROVE the moat fired was itself manufacturing false fires. I had the message tool loaded to warn the user "sorry for the spam" — reading `best()` first is the only reason I didn't broadcast a non-event. Fix: capture `maybeSurface`'s own return (with a local try/catch to keep the never-kill-the-tick posture), plus a regression test that spies `console.log` on a silent beat and asserts NO heartbeat line (the test the original lacked — it only ever exercised the send case).
**Why it matters:** (a) A monitoring/telemetry/health instrument is code and can have bugs, and a lying instrument is strictly worse than none: it manufactures false incidents (I nearly shipped a "fix" and disabled a healthy feature over a non-event), AND a permanently-firing signal MASKS the real event it exists to catch — a genuine spam loop would have been indistinguishable from the phantom, so the alarm was simultaneously crying wolf and blind. An instrument that's always-on is as useless as one that's always-off. (b) The specific, extremely common smell: conflating two different booleans — a generic error/transport wrapper's "did the call complete without throwing?" with a domain function's meaningful return "did the thing happen?". `best(fn)===true` and `await fn()===true` are different facts; wrapping a value-returning call in a success-only helper SILENTLY DISCARDS the value. The helper was correct for its other callers (fire-and-forget sends whose only question is "did it throw, so I can fall back?"); the bug was reusing it where the return carried meaning. Reuse that drops a return value is a type-checker blind spot (both are `Promise<boolean>`). (c) Verify-before-disclose / verify-before-fix: under a scary alarm the instinct is to alert the stakeholder and mitigate hard, but a false alarm ACTED ON is its own incident — alarming the user about nothing, or "fixing"/disabling a healthy system. The first move under an alarm is to verify the INSTRUMENT, especially a new one, with an INDEPENDENT measurement. Here a second source of truth (the diag process reading the real path, plus raw timestamped logs) disagreeing with the heartbeat localized the bug to the heartbeat in one read. (d) Mitigating was still right given the evidence at that moment (you can't tell from the logs alone that the instrument lies) — but the mitigation chosen was zero-cost-if-wrong (disabling a proactive feature during a window where it does nothing anyway). That's the correct shape: mitigate freely with reversible/idle-time actions, but gate DISCLOSURE and permanent fixes on verification.
**How to apply:** (1) Treat every observability instrument (heartbeat, health flag, metric, alert rule, telemetry counter) as testable code, and test BOTH directions — assert it does NOT fire when nothing happened, not only that it fires when something did. The silent/negative-case test is the one that catches a lying instrument; an instrument only ever tested on the positive path can be stuck-on and pass. (2) Never let a generic "did it succeed / not throw / retry-wrapper" helper stand in for a domain boolean that carries meaning — if a function returns a meaningful value, capture THAT value directly; reserve try/catch-success wrappers for fire-and-forget calls whose only question is "did it throw?". Wrapping a value-returning call in such a helper discards its meaning and the types won't warn you. Audit reuses of any `best()/safe()/tryRun()/withRetry()` wrapper for callees whose return value matters. (3) Under an alarm from an instrument, verify the instrument before disclosing the "incident" or shipping a fix: read the exact code path that emits the signal, and obtain an INDEPENDENT readout (a different process, a raw log, a second metric). When two independent sources disagree, suspect the newer/derived one. (4) Build observability with TWO independent readouts where you can (here: the heartbeat log AND a counts snapshot) — they cross-check each other, and their disagreement is often the fastest path to the bug. (5) Separate mitigation from disclosure: take zero-cost-if-wrong mitigations (disable an idle feature, add a guard) freely and immediately, but do NOT broadcast the incident to the stakeholder or land a permanent code change until you've verified the signal is real — a false alarm sent, or a healthy feature permanently "fixed," is a self-inflicted incident. Cross-project: any health/heartbeat/metric/telemetry/alerting/SLO instrument; any reuse of a generic error-handling/retry/success wrapper around a value-returning call; any incident response triggered by monitoring; the first firing of any newly-built instrument (assume it lies until a known-true and known-false case both check out).

## A new strategic claim that ABSORBS a mechanism from another doc must INHERIT that mechanism's own red-team caveats — and an independent adversarial pass catches the cross-document self-contradiction your own re-reading won't (2026-06-27)

A freshly-written ADR (ADR-066: "depth-legibility ≡ cold-start; the first session must produce a demonstrable depth-catch") named reality-drift as the *primary* session-1 proof, citing the SOURCE ADR's optimistic claim (ADR-064: "reality-drift solves cold-start, catchable at the moment of capture"). But a THIRD doc (COLD-START.md) had already red-teamed that same mechanism down to "a four-event lottery at n=1 until a calendar source exists." The new claim inherited the *ceiling* version and silently overrode the *corrected* version — a contradiction sitting in plain sight across my own document set, which I did NOT catch writing or re-reading ADR-066, because I was reasoning forward from ADR-064's framing.

**Why:** when you build a new claim on top of an existing mechanism, you tend to pull the mechanism's most-quotable (usually most-optimistic) framing, not its fully-caveated state — especially when the caveat lives in a *different* file than the one you're citing. Forward-reasoning from your own prior conclusion is exactly the blind spot an *independent* adversarial lens (a red-team subagent that re-reads the primary sources cold) is built to expose. It found the contradiction in one pass; I would not have.

**How to apply:** (1) before a new strategic claim steers a build/SPEC, run ONE independent red-team that re-reads the PRIMARY sources (not your synthesis) and is told to hunt for self-contradiction across the doc set. (2) When an ADR cites another ADR's mechanism, grep the rest of the corpus for that mechanism's name and check whether a *later/other* doc already corrected it — inherit the corrected state, not the headline. (3) Distrust "X solves Y" claims that are a mechanism's CEILING; ask "what's its reliability/floor, and did I already write that down somewhere else?" Ties to [[verify-the-instrument-before-you-disclose]] (verify before you act on a claim — here, verify the red-team's cited lines existed before integrating: they did).

## Recovery tooling must not depend on the thing it recovers; persist bootstrap-learned state to the source-of-truth; after dormancy, trust the LIVE clock (2026-07-02, Kept daemon cold-start incident)

The kept-daemon auto-deployer did a WARM swap: it lifted the secrets (bot token, API key) + the owner chat_id from the STILL-RUNNING container to carry them into the new one. Elegant for the common case, but it meant the deployer **could not revive a FULLY-DOWN daemon** — after a ~5-day host outage the container was gone, so there was nothing to lift from → `ABORT: 0/3 vars`. Worse: the owner chat_id was **bootstrap-learned** (read once from Tomi's `/start` via getUpdates) and lived ONLY in the container env — so a cold start lost it irrecoverably (it was in no file; the sandbox can't re-read it).

**Why it bites:** a restore path that reads its inputs FROM the live instance it's trying to restore is circular — it works only when restore isn't needed. And any required state that survives only in the ephemeral runtime is one crash away from being lost forever.

**How to apply:** (a) a recovery path must source its inputs from a **durable source-of-truth** (the secrets FILE / a persisted config), never from the running instance. (b) Any **bootstrap-learned / runtime-acquired state that is REQUIRED to function** must be persisted to that source-of-truth the moment it's learned, not left to survive only in the runtime env (fix shipped: `ops/cold-start.sh` writes the re-learned owner-id back into `.secrets.kept`). (c) **After a dormancy/gap, trust the LIVE clock** (`health.json.checked_at`), NOT the scheduled-task timestamps — a stale compaction summary + replayed "Jun 27" task timestamps had me operating ~5 days behind reality (as "gated on taste-gates" when the real state was "product DOWN"); the authoritative live artifact caught it. Ties to [[verify-the-instrument-before-you-disclose]] — verify current reality before acting on a stale mental model.
